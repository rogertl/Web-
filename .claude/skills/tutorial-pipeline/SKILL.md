# 通用教学流水线

## 触发条件

当用户提到以下任何内容时激活此 Skill：
- "开始阶段 X" / "生成阶段 X" / "继续生成"
- "生成所有阶段" / "一键生成" / "全部生成"
- "检查质量" / "重新评分"
- "生成练习" / "阶段练习"

## 核心原则

1. **技术栈无关** — 所有技术细节从 `SYLLABUS.md` 读取，Skill 本身是通用流水线
2. **质量守门** — 每个概念必须通过 5 维度评分才能输出
3. **可追溯** — 每次生成都有日志记录，支持复盘
4. **学员视角** — 内容面向"读懂 AI 代码"的实际需要

## 流水线流程

### 模式 A：单阶段生成（"开始阶段 X"）

```
对指定阶段 X 执行：

1. 读取阶段定义
   - 从 SYLLABUS.md 读取阶段 X 的概念清单和深度评分
   - 检查 README.md 中 checkbox 状态，跳过已完成的概念

2. 逐概念生成内容
   - 按 content-template.md 模板生成每个概念文件
   - 文件命名：{序号}-{概念名}.md（如 01-react-component.md）
   - 写入 stages/stage-X/ 目录

3. 质量评分（每个概念）
   - 按 quality-rubric.md 5 维度打分
   - 记录每个维度的分数和评语

4. 修订循环
   - 均分 < 4.0 → 自动修订，标注低分维度
   - 单维度 1 分 → 无论均分必须修订
   - 最多 2 轮修订
   - 2 轮后仍不达标 → 日志标记 "需人工审阅"

5. 阶段级练习
   - 生成 1 个综合练习（让 AI 生成对应小功能代码，学员讲解数据流）
   - 确保练习覆盖该阶段至少 80% 的概念

6. 更新 README
   - 将 stage-X/README.md 中的 checkbox 标记为已完成
   - 添加概念文件的链接

7. 记录日志
   - 按 log-template.md 格式追加到 logs/generation-log.md
   - 包含评分表、修订记录、练习信息

8. Git 提交
   - commit message: "feat: 生成阶段 X 教学内容"
```

### 模式 B：全自动一键生成（"生成所有阶段"）

```
1. 扫描所有阶段的 README.md
2. 跳过所有 checkbox 已完成（即所有概念已勾选）的阶段
3. 对每个未完成阶段，按模式 A 执行
4. 每个阶段独立 commit
5. 全部完成后输出总结报告
```

### 模式 C：质量检查（"检查质量"）

```
1. 读取指定阶段的已生成文件
2. 按 quality-rubric.md 重新评分
3. 输出评分报告
4. 如发现低分项，询问是否修订
```

## 参考文件

生成内容时引用以下文件（位于同目录下）：

| 文件 | 用途 |
|------|------|
| `quality-rubric.md` | 质量评分标准和修订规则 |
| `content-template.md` | 每个概念文件的结构模板 |
| `log-template.md` | 日志记录格式 |

## 关键路径

```
项目根目录/
├── SYLLABUS.md                          # 教学大纲（数据源）
├── stages/
│   └── stage-{0-6}/
│       ├── README.md                    # 进度跟踪（checkbox）
│       ├── {序号}-{概念名}.md           # 生成的教学内容
│       └── practice.md                  # 阶段综合练习（生成后添加）
├── logs/
│   └── generation-log.md               # 生成日志（追加模式）
└── .claude/skills/tutorial-pipeline/    # 本 Skill
```

## 注意事项

- **不要跳过已完成的概念** — 如果 README 中 checkbox 已勾选，说明该概念已生成且通过质量检查
- **不要硬编码技术内容** — 所有技术细节从 SYLLABUS.md 动态读取
- **每个概念文件保持独立** — 一个概念一个文件，便于单独阅读和修订
- **类比必须是生活化的** — 避免用技术概念类比技术概念
- **日志是追加模式** — 永远追加，不覆盖历史日志
