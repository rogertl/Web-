# 质量评分标准（Quality Rubric）

每个生成的概念文件必须通过 5 维度质量评分。每维度 1-5 分，均分 >= 4.0 方可通过。

---

## 评分维度

### 1. 概念清晰度（Concept Clarity）

| 分数 | 标准 |
|------|------|
| 5 | 一句话精准概括核心要点，零基础学员也能理解；标题和正文呼应紧密 |
| 4 | 核心概念清晰，但辅助说明稍多或稍少 |
| 3 | 核心概念基本讲清，但有冗余信息或关键信息遗漏 |
| 2 | 概念解释含糊，需要反复阅读才能理解 |
| 1 | 概念模糊，术语堆砌，学员看完仍然不知道这是什么 |

**检查点**：读一遍"一句话"部分，能否在不看正文的情况下理解这个概念是什么？

### 2. 类比有效性（Analogy Effectiveness）

| 分数 | 标准 |
|------|------|
| 5 | 类比精准、生活化、零基础可理解；对比表格清晰，一眼看出映射关系 |
| 4 | 类比恰当但稍抽象；或表格可以更精简 |
| 3 | 有类比但不够直观，需要额外解释才能理解 |
| 2 | 类比勉强相关，或用了技术概念类比技术概念 |
| 1 | 无类比，或类比误导（让学员产生错误理解） |

**检查点**：把类比拿给一个不懂技术的人看，他能理解吗？

### 3. 代码准确性（Code Accuracy）

| 分数 | 标准 |
|------|------|
| 5 | 代码语法正确、符合最佳实践；有上下文注释；有 AI 代码线索标记 |
| 4 | 代码正确但缺少注释或 AI 线索标记 |
| 3 | 代码基本正确但有小瑕疵（命名不规范、缺少 import 等） |
| 2 | 代码有语法错误或使用了过时的 API |
| 1 | 代码完全错误或误导学员 |

**检查点**：
- 代码能直接复制运行吗？
- 有没有标记"这段代码在 AI 生成项目中如何识别"？

### 4. 前置依赖连贯（Dependency Coherence）

| 分数 | 标准 |
|------|------|
| 5 | 严格按依赖链推进；每个新概念都有铺垫；引用的前置概念都在前面阶段讲过 |
| 4 | 基本连贯，偶有小跳跃但学员可以自行补上 |
| 3 | 有 1-2 处概念跳跃，需要额外说明 |
| 2 | 多处引用未讲过的概念，打乱学习节奏 |
| 1 | 大量引用后续概念，学员完全跟不上 |

**检查点**：列出文中提到的所有技术概念，检查每个是否在 SYLLABUS.md 的前置阶段中出现过。

**首次出现原则**：当文件中需要提到一个后续章节才深入讲解的概念时，必须用 1-2 句话简要解释"这是什么"，不能完全不提就使用。例如：提到 `npm run dev` 时，至少要说"dev 是 package.json 里定义的一个脚本名称，npm run 是执行它的命令"。后续章节再展开细节。

**文件内概念闭环**：文件中提到的每一个非日常概念，必须满足以下条件之一：
1. 在本文件内已有解释
2. 在前面阶段已完整讲解
3. 用 1-2 句话简要说明（首次出现原则），并标注"这在 X.X 会详细讲解"

三者都不满足 → 前置依赖连贯扣分。

### 5. 深度匹配度（Depth Matching）

| 分数 | 标准 |
|------|------|
| 5 | 精准匹配 SYLLABUS.md 中该概念的深度评分；不多不少 |
| 4 | 深度基本匹配，有轻微偏差 |
| 3 | 深度偏差明显（讲得太深或太浅） |
| 2 | 深度严重不匹配 |
| 1 | 完全忽略了深度要求 |

**检查点**：
- 深度 2-3 的概念：是否只讲"是什么"和"怎么用"，没有深入原理？
- 深度 4-5 的概念：是否讲透了原理、常见模式、注意事项？

---

## 修订规则

1. **均分 < 4.0** → 自动修订，标注具体低分维度，重点改进
2. **单维度 = 1** → 无论均分如何，必须修订该维度对应的内容
3. **修订轮次上限 = 2**，2 轮后仍不达标的记录到日志并标记"需人工审阅"
4. 修订后重新评分所有 5 个维度（不只是低分维度）

## 评分记录格式

```
| 概念 | 清晰度 | 类比 | 代码 | 依赖 | 深度 | 均分 | 状态 |
|------|--------|------|------|------|------|------|------|
| XX概念 | 5 | 4 | 5 | 4 | 5 | 4.6 | ✅ 通过 |
| YY概念 | 3 | 4 | 5 | 3 | 4 | 3.8 | 🔄 修订 |
```
