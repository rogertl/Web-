# 0.3 Vite：开发时的构建工具（热更新 HMR）

## 一句话

**Vite 让你改代码后浏览器立刻更新，不用手动刷新页面。**

## 为什么需要它

你写的代码不能直接在浏览器里运行——TypeScript 需要编译成 JavaScript，多个文件需要打包成少量文件，CSS 需要处理。这个"翻译+打包"的过程叫做**构建（Build）**。

没有构建工具：
1. 你写了一个组件（组件是 React 的基本单元，阶段 1 会详细讲）
2. 需要手动编译成浏览器能理解的 JS/CSS
3. 刷新浏览器看效果
4. 改一行代码 → 再编译 → 再刷新 → 再看

有了 Vite：
1. 你改了代码，保存
2. **保存的瞬间**，浏览器自动更新对应部分
3. 不用刷新，状态保留，即时看到效果

## 类比

| 概念 | 类比 |
|------|------|
| 没有 Vite | 改设计稿 → 重新印刷整本杂志 → 再翻到那一页 |
| 有 Vite | 改设计稿 → 那一页自动更新，其他页不动 |
| HMR | 菜里盐放多了，厨师直接换一份，不用重新点一桌菜 |

## 核心内容

### HMR 的工作原理

**HMR（Hot Module Replacement，热模块替换）** 是构建工具的核心能力。它的工作机制：

```
你修改代码并保存
    │
    ▼
Vite 检测到文件变化（监听文件系统）
    │
    ▼
只编译被修改的那个模块（不是整个项目）
    │
    ▼
通过 WebSocket（浏览器和服务器的实时通信通道）通知浏览器
    │
    ▼
浏览器只替换变化的模块（不刷新整个页面）
```

关键点：**只编译改动的部分，只替换改动的部分**。这就是为什么它快。

### Vite 为什么快

传统构建工具（如 Webpack，一个老牌的打包工具）在开发时会把所有文件打包成一个巨大的 JS 文件。项目越大，打包越慢。

Vite 的做法不同：
- **开发时**：不打包，利用浏览器原生的 ES Module（一种 JavaScript 模块加载标准）直接加载每个文件。改一个文件只需重新处理那一个文件
- **生产时**：才做完整打包优化（因为浏览器原生加载在上百个文件时会很慢）

### Vite vs Next.js 的构建工具

| | Vite | Next.js 内置构建工具 |
|---|---|---|
| 定位 | 纯前端开发构建工具 | 全栈框架，自带构建能力 |
| 适用场景 | 纯前端项目（SPA，即单页应用） | 需要服务端渲染的全栈项目 |
| AI 项目中 | 较少出现在 Next.js 项目中 | Next.js 项目默认使用 |

**关键**：Next.js 项目**不需要 Vite**，它有自己的构建工具（阶段 2 会详细讲）。但很多 AI 生成的纯前端项目（比如一个单独的 React 页面）会用 Vite。两个都认识即可。

## 你需要记住的

1. 看到 `vite.config.ts` → 这是一个 Vite 项目
2. 看到 `next.config.js` → 这是一个 Next.js 项目（不用 Vite）
3. HMR = 改代码后自动更新，原理是"只编译改动的部分 + WebSocket 通知浏览器"
4. Vite 快的原因：开发时不打包，利用浏览器原生能力逐文件加载
5. **构建工具只在开发时重要**，最终用户看到的是编译后的产物

## AI 代码中的线索

```js
// vite.config.ts — Vite 项目的标志文件
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
})
```

```js
// next.config.js — Next.js 项目的标志文件
/** @type {import('next').NextConfig} */
const nextConfig = {}
module.exports = nextConfig
```

打开项目根目录，看哪个配置文件存在，就知道用的什么构建体系。
